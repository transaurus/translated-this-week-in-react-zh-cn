# useSyncExternalStore - 被低估的React API

你可能听说过[`useSyncExternalStore()`](https://reactjs.org/docs/hooks-reference.html#usesyncexternalstore)，这是React 18推出的新hook，用于**订阅外部数据源**。状态管理库（如[Redux](https://github.com/reduxjs/react-redux/pull/1808)）常用它内部实现**选择器系统**。

但你是否想过在自己的应用代码中使用`useSyncExternalStore()`？

在这篇互动文章中，我想提出一个问题：**过度返回值的React hooks会触发不必要的重新渲染**。我们将看到`useSyncExternalStore()`如何成为解决方案。

## 过度返回的hooks

让我们用React-Router的[`useLocation()`](https://reactrouter.com/en/main/hooks/use-location)来演示这个问题。

该hook返回包含多个属性（`pathname`、`hash`、`search`等）的对象，但你可能不会用到所有属性。仅仅调用该hook就会在任意属性更新时触发重新渲染。

考虑以下应用：

```tsx
function CurrentPathname() {
  const { pathname } = useLocation();
  return <div>{pathname}</div>;
}

function CurrentHash() {
  const { hash } = useLocation();
  return <div>{hash}</div>;
}

function Links() {
  return (
    <div>
      <Link to="#link1">#link1</Link>
      <Link to="#link2">#link2</Link>
      <Link to="#link3">#link3</Link>
    </div>
  );
}

function App() {
  return (
    <div>
      <CurrentPathname />
      <CurrentHash />
      <Links />
    </div>
  );
}
```

<App />

每次点击hash链接时，`CurrentPathname`组件都会重新渲染——即使它根本没有使用`hash`属性😅。

:::tip

当hook返回的数据未被使用时，请考虑React的重新渲染机制。如果不加注意，在React树顶部添加的一个小小`useLocation()`调用就可能损害应用性能。

:::

:::info

本文目的不是批评React-Router，而是阐明问题。`useLocation()`只是用来构建这篇互动文章的典型案例。你自己的React hooks和其他第三方库也可能存在过度返回值的问题。

:::

## `useSyncExternalStore`来救援？

[官方文档](https://reactjs.org/docs/hooks-reference.html#usesyncexternalstore)说明：

> useSyncExternalStore是推荐用于从外部数据源读取和订阅的hook，其实现方式与并发渲染特性（如选择性水合和时间切片）兼容。
> 该方法返回存储的值并接受三个参数：
>
> - `subscribe`: 注册回调的函数，在存储变化时调用
> - `getSnapshot`: 返回存储当前值的函数
> - `getServerSnapshot`: 返回服务端渲染时所用快照的函数

```tsx
function useSyncExternalStore<Snapshot>(
  subscribe: (onStoreChange: () => void) => () => void,
  getSnapshot: () => Snapshot,
  getServerSnapshot?: () => Snapshot,
): Snapshot;
```

这听起来有些抽象。[Beta文档页面](https://beta.reactjs.org/learn/you-might-not-need-an-effect#subscribing-to-an-external-store)给出了很好的示例：

```tsx
function subscribe(callback) {
  window.addEventListener("online", callback);
  window.addEventListener("offline", callback);
  return () => {
    window.removeEventListener("online", callback);
    window.removeEventListener("offline", callback);
  };
}

function useOnlineStatus() {
  return useSyncExternalStore(
    subscribe,
    () => navigator.onLine,
    () => true,
  );
}

function ChatIndicator() {
  const isOnline = useOnlineStatus();
  // ...
}
```

实际上，浏览器历史记录也可以视为外部数据源。让我们看看如何结合React-Router使用`useSyncExternalStore`！

## 实现`useHistorySelector()`

React-Router提供了我们连接`useSyncExternalStore`所需的一切：

- 通过[`useHistory()`](https://v5.reactrouter.com/web/api/Hooks/usehistory)访问浏览器历史记录
- 通过[`history.listen(callback)`](https://github.com/remix-run/history/blob/main/docs/api-reference.md#history.listen)订阅历史更新
- 通过[`history.location`](https://github.com/remix-run/history/blob/main/docs/api-reference.md#historylocation)获取当前位置快照

:::caution

本网站使用React-Router v5：React-Router v6的解决方案会有所不同（[参见](https://twitter.com/Zh0uzi/status/1567523679604539405)）。

:::

`useHistorySelector()` 的实现相对简单：

```tsx
function useHistorySelector(selector) {
  const history = useHistory();
  return useSyncExternalStore(history.listen, () =>
    selector(history),
  );
}
```

让我们在应用中使用它：

```tsx
function CurrentPathname() {
  const pathname = useHistorySelector(
    (history) => history.location.pathname,
  );
  return <div>{pathname}</div>;
}

function CurrentHash() {
  const hash = useHistorySelector(
    (history) => history.location.hash,
  );
  return <div>{hash}</div>;
}
```

<AppFixed />

现在，当你点击上方的哈希链接时，`CurrentPathname` 组件将**不再重新渲染**！

## 另一个示例：`scrollY`

我们可以订阅的外部数据源非常多，在其之上实现自己的选择器系统可能有助于优化 React 的重新渲染。

例如，假设我们想使用页面的 `scrollY` 位置。我们可以实现这个自定义 React 钩子：

```tsx
// A memoized constant fn prevents unsubscribe/resubscribe
// In practice it is not a big deal
function subscribe(onStoreChange) {
  global.window?.addEventListener("scroll", onStoreChange);
  return () =>
    global.window?.removeEventListener(
      "scroll",
      onStoreChange,
    );
}

function useScrollY(selector = (id) => id) {
  return useSyncExternalStore(
    subscribe,
    () => selector(global.window?.scrollY),
    () => undefined,
  );
}
```

现在我们可以使用这个钩子，并传入一个可选的选择器：

```tsx
function ScrollY() {
  const scrollY = useScrollY();
  return <div>{scrollY}</div>;
}

function ScrollYFloored() {
  const to = 100;
  const scrollYFloored = useScrollY((y) =>
    y ? Math.floor(y / to) * to : undefined,
  );
  return <div>{scrollYFloored}</div>;
}
```

<ScrollApp />

滚动页面并观察上方组件的重新渲染情况？其中一个组件的重新渲染次数比另一个少！

:::info

当你不需要 `scrollY` 的 1 像素精度级别时，返回一个宽泛的值（如 `scrollY`）也可以被视为过度返回。考虑返回一个更窄的值。

例如：一个仅返回有限值集合（`small`、`medium` 或 `large`）的 `useResponsiveBreakpoint()` 钩子，会比返回 `useViewportWidth()` 钩子更优化。

如果 React 组件仅对 `large` 屏幕有不同处理，你甚至可以创建一个更窄的 `useIsLargeScreen()` 钩子，返回一个布尔值。

:::

## 结论

我希望这篇文章能让你重新审视 `useSyncExternalStore()`。我觉得这个钩子在 React 生态系统中目前使用不足，值得更多关注。有许多外部数据源可以订阅。

如果你还没有升级到 React 18，可以使用 npm 上的 [use-sync-external-store](https://www.npmjs.com/package/use-sync-external-store) 垫片，它已经支持旧版本。如果你需要返回一个记忆化的非原始值，还可以使用 `use-sync-external-store/with-selector` 导出。
