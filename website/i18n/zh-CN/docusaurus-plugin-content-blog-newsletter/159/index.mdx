## ⚛️ React {#react}

[![Bun 1.0](/emails/issues/159/bun.jpg)](https://bun.sh/blog/bun-v1.0)

[**Bun 1.0**](https://bun.sh/blog/bun-v1.0)

本周不可能错过它：Bun 刚刚发布了稳定的 v1.0 版本。这个消息虽然不直接与 React 相关，但无疑会在某个时候影响到所有 React 开发者。

Bun 是一个基于 Webkit 并用 Zig 编写的完整 JavaScript 和 TypeScript 工具链。它是我们今天使用的众多前端工具的极速替代品：Node.js、npx、Babel、esbuild、swc、webpack、Jest、Vitest、npm、Yarn、pnpm... 基准测试结果令人印象深刻且绝对真实。

它与现有工具的兼容性以及对混合 ESM/CJS 的支持使其很容易作为即插即用的替代品。Bun 通常能够直接运行你的代码而无需任何修改（否则会被视为 [🐦 一个错误](https://twitter.com/jarredsumner/status/1700497869160268155)）。React 框架如 Next.js、Remix 和 Astro 已经支持它。将其作为 npm/Yarn/pnpm 或 Jest/Vitest 的替代品可能是一个不错的切入点，风险也不大。

## 📱 React-Native {#react-native}

[![静态版Hermes](/emails/issues/159/hermes.jpg)](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement)

[**静态版Hermes**](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement)

静态版Hermes在上周的React-Native EU大会上发布。我未亲临现场，因此通过阅读幻灯片尝试理解其原理，目前相关视频尚未上传至YouTube。

静态版Hermes是一个实验性项目，允许您将部分TypeScript（或Flow）代码编译为原生代码。相比基于字节码解释的常规Hermes模式，这能带来显著的性能提升（10-20倍）。同时还能直接从TypeScript代码轻松集成原生API（零成本FFI，比JSI封装快15-80倍）。

要实现这一点，TypeScript首先需成为"健全"语言。静态版Hermes将修改JavaScript语义，确保声明类型与运行时类型匹配。这张幻灯片很好地阐释了该概念：

[![静态版Hermes使TypeScript变得健全](/emails/issues/159/hermes2.jpg)](https://speakerdeck.com/tmikov2023/static-hermes-react-native-eu-2023-announcement?slide=14)

这个非常有趣的构想，或许有朝一日能对React-Native生态系统之外产生影响？
