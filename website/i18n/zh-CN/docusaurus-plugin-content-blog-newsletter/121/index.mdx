## ⚛️ React {#react}

**[为什么我们要放弃CSS-in-JS](https://dev.to/srmagura/why-were-breaking-up-wiht-css-in-js-4g9b)**

Emotion维护者Sam分析了CSS-in-JS的利弊，重点指出运行时库的问题。对他们公司Spot而言，样式序列化的成本过高，最终用Sass模块（CSS模块+Sass）和CSS工具系统替代了Emotion。我们在Docusaurus中使用CSS模块，可以确认这是个不错的无运行时方案，虽然开发体验稍逊但仍可接受。

这篇文章引发了热烈讨论。其实CSS-in-JS运行时库的性能问题早有共识。[Dan Abramov证实](https://twitter.com/dan_abramov/status/1584835028986793984)并[表示不会在新版React文档站点使用任何运行时](https://github.com/reactjs/reactjs.org/issues/5115)。Ryan Florence提到[使用Emotion的Chakra存在性能问题](https://twitter.com/ryanflorence/status/1582835729360572416)。

Tailwind仍是不错的替代方案。在我看来，它可归类为无运行时CSS-in-JS方案，特别是在React中通过JSX应用Tailwind样式。虽然有人质疑其[可维护性](https://twitter.com/hi__mayank/status/1584261503670448128)，但实际效果很好。

**[Qwik React](https://qwik.builder.io/qwikcity/integrations/react/)**

Qwik是一款创新的SPA框架，[避免了与客户端水合相关的性能问题](https://www.builder.io/blog/hydration-is-pure-overhead)。现在它可以通过`qwikify$()`函数与React集成，用于包装任何现有的React组件。`client:`指令（灵感来自Astro）可以精细控制React组件的水合时机。演示中集成了Material UI到Qwik应用中。另可参阅[此视频](https://www.youtube.com/watch?v=fLH28rDu97I)。我认为这是避免全盘投入这个新框架的好方案，可以逐步尝试。在保持React跨平台特性的同时，采用Qwik来提升Web端性能。

**[全类型化Web应用](https://www.epicweb.dev/fully-typed-web-apps)**

Kent C. Dodds阐述了实现端到端类型安全的价值——从数据库到UI，跨越网络、本地存储、数据库等不同边界。他讨论了验证（模式、类型守卫）、类型生成（Prisma）和约定（Remix），并解释如何通过类型推断在Remix中实现这种体验：使用`typeof`比`as`更可取。

**[在Next.js中构建交互式WebGL体验](https://vercel.com/blog/building-an-interactive-webgl-experience-in-next-js)**

Vercel揭秘了他们如何基于React-Three-Fiber为[Next.js大会打造沉浸式注册页面](https://nextjs.org/conf)。对于没有3D/WebGL/Three.js经验者仍较难理解，但分步拆解的方式展现了创意开发的思维方式。

- 📜 [微软 - 扩展 Azure 静态 Web 应用对 Next.js 的支持](https://techcommunity.microsoft.com/t5/apps-on-azure-blog/extending-next-js-support-in-azure-static-web-apps/ba-p/3627975)：Azure 将现支持 Next.js SSR（此前仅支持 SSG）。
- 📜 [Remix 中表单提交后保留输入值](https://sergiodxa.com/articles/persist-inputs-after-a-form-submit-in-remix)：提出渐进增强方案以保留表单提交后的值。两种方案：在 Remix action 中返回值或使用闪存会话。

- 📜 [Remix 中抛出异常与返回响应对比](https://sergiodxa.com/articles/throwing-vs-returning-responses-in-remix)
