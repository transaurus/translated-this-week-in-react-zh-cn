## ⚛️ React动态 {#react}

**[为什么我们要放弃CSS-in-JS](https://dev.to/srmagura/why-were-breaking-up-wiht-css-in-js-4g9b)**

Emotion维护者Sam分析了CSS-in-JS的利弊，重点指出运行时库的问题。对其公司Spot而言，样式序列化的成本过高。他们用Sass模块（CSS模块+Sass）和CSS工具系统替代了Emotion。我们在Docusaurus中使用CSS模块，可以证实这是种不错的无运行时替代方案，虽然开发体验稍逊但仍可接受。

此文引发广泛讨论。其实CSS-in-JS运行时导致的性能问题早有共识。[Dan Abramov证实](https://twitter.com/dan_abramov/status/1584835028986793984)新版React文档站点将[弃用运行时方案](https://github.com/reactjs/reactjs.org/issues/5115)。Ryan Florence提到[Chakra使用Emotion的性能问题](https://twitter.com/ryanflorence/status/1582835729360572416)。

Tailwind仍是不错的替代方案。我认为在React语境下，通过JSX实现的Tailwind样式可归类为无运行时CSS-in-JS。虽有人质疑其[可维护性](https://twitter.com/hi__mayank/status/1584261503670448128)，但实际效果良好。

**[Qwik React](https://qwik.builder.io/qwikcity/integrations/react/)**

Qwik 是一款创新的单页应用框架，[避免了与客户端水合相关的性能问题](https://www.builder.io/blog/hydration-is-pure-overhead)。现在它可以通过 `qwikify$()` 函数与 React 集成，用于包装任何现有的 React 组件。`client:` 指令（灵感来自 Astro）允许以极高的粒度控制 React 组件的水合时机。该演示将 Material UI 集成到 Qwik 应用中，[详见此视频](https://www.youtube.com/watch?v=fLH28rDu97I)。我认为这是避免全盘投入这一新框架、逐步尝试的好方案。在保持 React 跨平台特性的同时，也能通过 Qwik 提升 Web 端性能。

**[全类型化 Web 应用](https://www.epicweb.dev/fully-typed-web-apps)**

Kent C. Dodds 阐述了实现端到端类型安全的价值——从数据库到用户界面，跨越网络、本地存储、数据库等不同边界。他探讨了验证（模式、类型守卫）、类型生成（Prisma）和约定（Remix），并解释如何通过类型推断在 Remix 中实现这一体验：使用 `typeof` 比 `as` 更可靠。

**[在 Next.js 中构建交互式 WebGL 体验](https://vercel.com/blog/building-an-interactive-webgl-experience-in-next-js)**

Vercel 分享了他们如何基于 React-Three-Fiber 为 [Next.js 大会打造令人印象深刻的沉浸式注册页面](https://nextjs.org/conf)。对于没有 3D/WebGL/Three.js 经验的开发者而言仍较复杂，但分步拆解的方式展现了创意开发的思维方式，颇具启发性。
