## ⚛️ React技术栈 {#react}

[![为何客户端组件需要SSR生成HTML？](/emails/issues/147/client-component.jpg)](https://github.com/reactwg/server-components/discussions/4)

**[为何客户端组件需要SSR生成HTML？](https://github.com/reactwg/server-components/discussions/4)**

Dan Abramov用可视化方式解析客户端树（传统React组件）与服务端树（新概念）的区别。这张示意图很有价值，它澄清了一个常见误解：客户端组件本质上就是我们熟知的普通React组件。在SSR过程中，**客户端组件其实会在服务端执行**，与从前完全一致。

[![Raycast API与插件工作原理](/emails/issues/147/raycast.jpg)](https://www.raycast.com/blog/how-raycast-api-extensions-work)

**[Raycast API与插件工作原理](https://www.raycast.com/blog/how-raycast-api-extensions-work)**

Raycast 是一款可扩展的 macOS 启动器，以其卓越品质著称的应用。本文阐述了他们实现插件系统的技术路径。团队不愿在用户体验和开发者体验上妥协：既未采用 WebView 也不使用 Electron，在确保安全性的同时支持 npm 包调用。最终他们选择开发定制化的 React 渲染器，该渲染器运行在独立的 Node 进程中，通过 JSON 桥接协议驱动 Swift 原生视图——这套架构与 React-Native 的设计理念高度相似。

- 📖 [Next.js文档更新 - 项目组织页面](https://twitter.com/delba_oliveira/status/1662211824211877889)
- 📜 [React是否正在经历Angular.js时刻？](https://marmelab.com/blog/2023/06/05/react-angularjs-moment.html): 服务端组件的出现是否类似于Angular.js到Angular 2的范式转变？答案是否定的，因为服务端组件是一个可选的新特性，React的其他部分仍保持原有功能。不过文章也提醒，现在推荐和采用RSC可能还为时过早，生态系统尚未完全准备好。
- 📜 [关于React服务端组件的观点笔记](https://github.com/nilshartmann/react-server-component-material/blob/notes-on-rsc/notes-on-rsc.md): 一系列关于RSC的笔记和思考。内容全面且文档丰富，包含许多相关的外部参考。
- 📜 [同构开发](https://meijer.ws/articles/isomorphic-development): 关于服务端组件带来的复杂性的有趣思考。复杂之处不在于服务端组件本身，而在于代码库的新混合性质及其边界的不明确性。
- 📜 [React API和代码复用的演变](https://frontendmastery.com/posts/the-evolution-of-react-patterns/): 回顾了不同时期使用的模式及其权衡（混入、高阶组件、渲染属性、钩子、RSC）。
- 📜 [为什么我们放弃支持Next.js](https://www.stormkit.io/blog/why-we-are-dropping-support-for-next-js): Stormkit部署平台放弃了对Next.js的支持，因为对于他们的小团队来说过于复杂。相反，他们提出了一个支持SSR的React + Vite模板。
- 📜 [使用服务端组件的更好React学习路径](https://scastiel.dev/better-learning-path-react): 如果RSC实际上让React更容易学习呢？我认为从新的角度来看是有道理的。
- 📜 [RedwoodJS的下一个时代：全面拥抱React服务端组件](https://tom.preston-werner.com/2023/05/30/redwoods-next-epoch-all-in-on-rsc.html): Tom Preston-Werner（GitHub创始人）宣布Redwood框架计划实现React服务端组件。此前，Redwood生成的是没有SSR的SPA（对SEO有限制），并且强制使用GraphQL并不总是最佳选择。
- 📜 [解决Remix中的水合错误](https://www.jacobparis.com/content/remix-hydration-errors): 关于调试React水合错误的实用建议。这些建议也适用于其他框架。
- 📜 [什么是JSX.IntrinsicElements？](https://www.totaltypescript.com/what-is-jsx-intrinsicelements): TypeScript使用IntrinsicElements来确定哪些JSX元素是原生支持的。可以声明自己的原生元素，这正是像React这样的框架所做的。
- 📜 [使用SST在边缘部署Next.js](https://engineering.udacity.com/deploying-next-js-on-the-edge-with-sst-is-sst-the-game-changer-its-claimed-to-be-1f05a0abc27c): 关于使用OpenNext和SST在AWS上部署Next.js应用的积极反馈。
- 📜 [Spotify - 设计系统中的多层抽象](https://engineering.atspotify.com/2023/05/multiple-layers-of-abstraction-in-design-systems/): Spotify解释了他们的React设计系统如何暴露三个抽象层次（配置、插槽、自定义）。
