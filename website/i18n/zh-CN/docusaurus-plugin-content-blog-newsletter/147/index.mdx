## ⚛️ React 动态 {#react}

[![为何客户端组件需要 SSR 成 HTML？](/emails/issues/147/client-component.jpg)](https://github.com/reactwg/server-components/discussions/4)

**[为何客户端组件需要 SSR 成 HTML？](https://github.com/reactwg/server-components/discussions/4)**

Dan Abramov 用可视化方式解析了客户端树（传统模式）与服务端树（新模式）的区别。这张图表颇具启发性：客户端组件本质上就是传统 React 组件，在 SSR 过程中**它们仍会在服务端执行**，与从前完全一致。

[![Raycast API 与插件运作机制](/emails/issues/147/raycast.jpg)](https://www.raycast.com/blog/how-raycast-api-extensions-work)

**[Raycast API 与插件运作机制解析](https://www.raycast.com/blog/how-raycast-api-extensions-work)**

Raycast 是一款可扩展的 macOS 启动器，这款应用以卓越品质著称。本文详细阐述了他们实现插件系统的技术路径。团队在用户体验(UX)和开发者体验(DX)上拒绝妥协：不使用 WebView 或 Electron，确保安全性，同时支持 npm 包调用。最终方案是开发了一个运行在独立 Node 进程中的自定义 React 渲染器，通过 JSON 桥接驱动 Swift 原生视图——其架构与 React-Native 极为相似。

- 🥳️ [React十周年庆典](https://vercel.com/blog/10-years-of-react): Vercel通过发布[圆桌讨论视频](https://www.youtube.com/watch?v=g5BGoLyGjY0)来庆祝React诞生十周年，Andrew、Sebastian和Delba共同探讨React最新创新成果。

- 👀 [Remix + RSC概念验证](https://twitter.com/ryanflorence/status/1663587026384584704): Remix终于计划实现服务端组件，并在此[推文](https://twitter.com/remix_run/status/1661017634480201734)中给出了原因之一。
- 💬 [React工作组 - 从零实现RSC. 第一部分：服务端组件](https://github.com/reactwg/server-components/discussions/5): Dan Abramov深入讲解如何从头实现服务端组件。第一部分（共三部分）已发布，详细描述服务端组件的实现方式。
- 📖 [Next.js文档更新 - 项目组织页面](https://twitter.com/delba_oliveira/status/1662211824211877889)
- 📜 [React是否正在经历Angular.js时刻？](https://marmelab.com/blog/2023/06/05/react-angularjs-moment.html): 服务端组件的出现是否像Angular.js到Angular 2那样的范式转变？文章认为不是，因为服务端组件是可选新特性，原有React功能仍可正常使用。但作者也警告现阶段全面采用RSC可能为时过早，生态系统尚未完全成熟。
- 📜 [关于React服务端组件的观点笔记](https://github.com/nilshartmann/react-server-component-material/blob/notes-on-rsc/notes-on-rsc.md): 对RSC的全面思考与记录，包含大量相关外部参考资料。
- 📜 [同构开发](https://meijer.ws/articles/isomorphic-development): 探讨服务端组件带来的复杂性。真正的复杂性不在于RSC本身，而在于代码库新的混合特性及其模糊的边界。
- 📜 [React API与代码复用的演进](https://frontendmastery.com/posts/the-evolution-of-react-patterns/): 回顾不同时期使用的模式及其权衡（混入、高阶组件、渲染属性、钩子、RSC）。
- 📜 [为何我们放弃支持Next.js](https://www.stormkit.io/blog/why-we-are-dropping-support-for-next-js): Stormkit部署平台因Next.js过于复杂而放弃支持，转而推荐使用支持SSR的React+Vite模板。
- 📜 [结合服务端组件的React学习路径优化](https://scastiel.dev/better-learning-path-react): 如果以全新视角看待，RSC可能使React更易学习。
- 📜 [RedwoodJS新时代：全面拥抱React服务端组件](https://tom.preston-werner.com/2023/05/30/redwoods-next-epoch-all-in-on-rsc.html): GitHub创始人宣布Redwood框架将实现RSC。此前Redwood生成无SSR的SPA（不利于SEO），且强制使用GraphQL并非总是最佳选择。
- 📜 [解决Remix中的水合错误](https://www.jacobparis.com/content/remix-hydration-errors): 调试React水合错误的实用建议，同样适用于其他框架。
- 📜 [什么是JSX.IntrinsicElements？](https://www.totaltypescript.com/what-is-jsx-intrinsicelements): TypeScript通过IntrinsicElements识别原生支持的JSX元素。框架如React可声明自定义原生元素。
- 📜 [使用SST在边缘部署Next.js](https://engineering.udacity.com/deploying-next-js-on-the-edge-with-sst-is-sst-the-game-changer-its-claimed-to-be-1f05a0abc27c): 关于使用OpenNext和SST在AWS部署Next.js的积极实践。
- 📜 [Spotify设计系统中的多层抽象](https://engineering.atspotify.com/2023/05/multiple-layers-of-abstraction-in-design-systems/): 阐述其React设计系统如何提供三个抽象层级（配置、插槽、自定义）。
- 📦 [TypeScript 5.1](https://devblogs.microsoft.com/typescript/announcing-typescript-5-1): 新版本对React意义重大，改进了JSX类型支持，允许组件返回Promise（异步服务端组件）、字符串、数组等。新版React类型定义已[发布](https://twitter.com/sebsilbermann/status/1664356039876124672)。
