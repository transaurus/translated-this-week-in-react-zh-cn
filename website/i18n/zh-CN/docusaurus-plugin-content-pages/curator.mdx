# This Week In React - 内容策展人招募

我最初创建[This Week In React](https://thisweekinreact.com/)是作为一个帮助React开发者了解生态动态的精选资讯简报。

长期目标是将其打造成真正由社区驱动的React和React Native开发者媒体平台（类似CSS-Tricks），而不仅是一份每周链接列表邮件。

为此需要团队协作，现招募简报合作者共同制作每周内容。

这是**正式工作**：有薪酬，要求**高度可靠**，且愿意为每期简报投入**14-18小时工作量**。

## 对This Week In React的益处

本次招募的主要目标：

- 建立可靠备份机制，避免因休假导致的倦怠
- 通过每年50+期内容打造专业媒体形象
- 增强社区驱动属性
- 通过作者轮换提升内容多样性
- 腾出时间进行媒体平台深度开发
- 维持简报内容的高质量标准

过去五年我几乎独自运营，初期乐趣可能逐渐变成长期负担。我希望能自主选择而非被迫发布内容，最终实现项目能脱离我个人持续运作。

## 客座策展人收益

您将获得：

- 每期简报的**合理报酬**
- 提升**个人品牌**及公司曝光度（[客座策展人案例](https://thisweekinreact.com/newsletter/206)）
- 回馈社区，参与有价值项目

若有创新提议，欢迎探讨双赢方案。

## 工作流程

**简而言之**：每周一启动内容筛选，周三傍晚（约UTC17点/PST上午9点）发送简报。你需确保邮件在周三前完全就绪。通常总计需要约16小时工作量，可由2名开发者并行处理。

**主要步骤**：

- 提前数周规划负责的期次
- 通过Twitter、RSS及自然渠道初步筛选内容
- 将候选链接添加至协作Google文档
- 复审链接，格式化并剔除低价值内容
- 对优质链接添加专业点评
- 选定头条新闻
- 挑选表情包（最好与当周新闻相关）
- 拟定含React/RN关键词的标题
- 撰写导语
- 转换为MDX格式
- 推送至Git并发布在线版
- 在社交平台协助推广

**由我保留的职责**：

- 管理赞助商及广告位
- 将网页版转换为邮件格式并发送

**待定事项**：

- 发布Twitter推文或文章
- 在Bluesky、Reddit、LinkedIn等平台分发

提前数月查看[公开日历](https://thisweekinreact.com/calendar)确定负责期次。确认日期后，你将成为该期指定作者，**必须预留约2个工作日**进行处理。

## 编辑方针

**核心理念**：

- 面向**中高级React/React Native开发者**，但不限于React相关内容
- **新鲜内容**。超过1个月的旧闻通常已覆盖
- **前沿动态**，预见技术发展趋势
- **事件驱动型内容**，如新版本发布、新项目、文档更新
- 成为**生态脉搏**，而非单纯的React npm包发现工具

**优先考虑**：

- 聚焦于**单一理念**并**深度**阐释某个概念的优质内容
- 专注于**React核心技法**、能被广大开发者普遍应用的内容
- 呈现**新颖**、**创新**或**被低估**技术方案的内容
- 当呈现方式出色或影响深远时，涵盖**细分领域**和**弱信号**的内容
- 具有**卓越呈现力**和**绝佳阅读体验**的内容，例如交互式技术文章

**需规避的内容：**

- 因SEO问题需过滤AI生成内容、清单式文章及其他低质量内容
- 过滤存在付费墙、需登录认证或阅读体验差的内容
- 过滤面向初级开发者的基础教程、步骤指南、对官方文档无价值重述的内容
- 过滤教授付费产品使用的教程（我们有赞助渠道处理这类需求），除非其非过度营销且能同步传授有价值的React核心技术
- 过滤大多数UI库和样板代码。例外情况是：当它们展现出高关注度、长期维护承诺或出自权威来源时。我们通常不报道这些工具的"首发"，但可能在v10版本成为主流并证明社区价值后进行报道
- 过滤过于微弱的弱信号：若您自身都难以理解该内容、无法用简练语言解释其影响、或认为仅少数人受益，则可能时机未到或价值不足
- 尽可能避免X/Twitter话题链接（需登录）。优先选用站外链接。但包含完整信息、外链或视频演示的独立推文仍可选用

对于呈现形式卓越的内容可破例采纳。例如解释React基础概念的交互式文章：这是目标受众（资深开发者）可能分享给初级同事的优质资源

## 内容规范

鉴于React生态的繁荣，简报会包含大量链接。
我们不是通过更严格过滤链接，而是优化高信息密度下的阅读体验。
内容应保持一致性、便于速览，并尽可能采用"内联"信息呈现方式。
要避免诱导点击却令人失望的链接。
即使读者不点击任何链接，阅读简报本身也应获得价值。

### 板块划分

简报包含5个主要板块：

- 导读：本周要闻摘要
- React板块：面向所有React开发者及Web开发者的相关内容
- React Native板块：面向React Native开发者的相关内容
- 其他板块：面向更广泛JS受众的内容，含TS/CSS/TC39/HTML/工具链等
- 趣味板块：1-3个meme或趣味截图，优先选择与React（内部梗）及本周新闻相关的素材

### 头条选取

常规选择标准：

- 1条React头条
- 1条React Native头条

每条头条通常配有顶部横幅图。
优先选择具有视觉意义的内容（如代码截图），精美的社交卡片图亦可接受。

有时某个板块没有明确头条，有时则可能出现双头条。
偶尔我们会为"其他"板块选取头条。

### 格式规范

所有链接遵循基本格式：

- \<表情符号\> [链接标题](https://example.com)：可选的编者主观评注

若原标题过长、存在点击诱导或大小写混乱，可重构标题以保持整体一致性

对于版本发布，我们省略"v"前缀和补丁版本号（例如3.1，而非v3.1.9）。在链接标题中用短横线`-`分隔主要版本的关键特性。若补丁版本没有值得关注的更新（通常如此），则不予收录。链接内的内容是"客观"部分（特性说明），链接外的则是您的主观评论（如有）。示例：

- 📦 [Astro 5.2 - Tailwind 4支持、尾部斜杠重定向、禁用React流式渲染选项](https://astro.build/blog/astro-520/)：流式渲染可能与运行时CSS-in-JS库不兼容
- 📦 [Nextra 4.0 - 现已采用App Router](https://the-guild.dev/blog/nextra-4)
- 📦 [React Scan 0.1.2 - 重渲染历史记录、组件树](https://github.com/aidenybai/react-scan/releases/tag/v0.1.2)

若是首次收录的新项目，可省略版本号，用客观的项目描述替代关键词（参考GitHub仓库说明、落地页或SEO标签）：

- 📦 [Polygen - 在应用中静态运行WebAssembly模块，无需解释器](https://github.com/callstackincubator/polygen)：Callstack的预览版工具，通过将Wasm转为C代码并通过JSI调用以实现卓越性能
- 📦 [React Three Handles](https://pmndrs.github.io/xr/docs/handles/introduction)：构建React XR交互的新工具包
- 📦 [mdx-prompt - 使用JSX组合式LLM提示词](https://edspencer.net/2025/2/3/mdx-prompt-composable-prompts-with-jsx)

这是我们常用的表情符号分类：

- 💸：赞助内容
- 👀：预览、PR、弱信号
- 💬：RFC、社区讨论
- 📊：调研、基准测试
- 💡：有趣创意
- 📖：文档
- 🦋：Bluesky链接
- 🐦：Twitter/X链接（尽量避免）
- 🎨：视觉演示（非X平台发布）
- 📅：活动、会议合作
- 📜：文章
- 📦：版本发布、npm包/插件/代码库
- 🔗：资源
- 🎥：视频
- 🎙️：播客

我们通常遵循以下"表情符号排序"，形成一致的阅读顺序：

- 头条新闻
- 弱信号
- 社区文章
